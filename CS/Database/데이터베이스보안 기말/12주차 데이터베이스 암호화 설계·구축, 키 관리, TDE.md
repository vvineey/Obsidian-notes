
# 1. 데이터베이스 암호화 생명 주기 

**데이터베이스 암호화 단계**
```
암호화 설계 → 암호화 구축 → 암호화 운영
```

|단계|주요 작업|
|---|---|
|설계|사용자 식별, 암호화 대상·알고리즘 선정, 권한 설계|
|구축|실제 데이터 암호화, 제약사항 유지, 키 관리체계 구축|
|운영|키 변경·백업·폐기, 접근통제, 감사, 성능 관리|



# 2. 데이이터베이스 암호화 설계 

데이터베이스 암호화를 구축하기 전 필요 사항을 식별하고 계획하는 단계

**설계항목**
- 데이터베이스 사용자 식별 
- 비인가 사용자 통제
- 복호화 권한 통제
- 암호화 대상 선정 
- 암호화 알고리즘 선정



# 3. 데이터베이스 사용자 식별 

암호화를 수행하기 전에 누가 데이터베이스를 사용하며, 어떤 방법으로 접근하는지 파악하는 단계 

**사용자 식별 정보**
- 사용자 계정
- IP 주소 
- 접속 날짜 시간
- 접속 도구 
- 접근 데이터 
- 데이터 변경 가능 여부 등등

**정형적인 접근**
특정 업무를 위해 정기적이고 반복적으로 데이터베이스에 접근하는 방식 
일반적으로 미리 정해진 SQL을 반복적으로 수행한다.

ex) 웹 서버, WAS, 업무 응용프로그램 등


**비정형적인 접근**
DBMS 유지보수나 개발을 위해 사용자가 데이터베이스에 직접 접근하는 방식 
다양한 SQL을 직접 실행하고 데이터를 조회·수정·삭제할 수 있으므로 더 강력한 접근통제와 감사가 필요하다.

ex) DBA, 개발자 등



# 4. 비인가 사용자와 복호화 권한 통제 

사용자의 권한에 따라 표시되는 데이터 형태를 다르게 설계해야 한다.

```
인가 사용자 → 복호화된 원본 데이터 표시
비인가 사용자 → 암호문, NULL, 마스킹 데이터 표시
```


# 5. 암호화 대상 선정 

암호화는 보안성을 높이지만 처리 성능과 운영 복잡성에 영향 -> 민감도, 법적 요구사항, 성능을 고려 

**암호화 대상**
- 컬럼 암호화 
- 데이터베이스 전체 암호화 


# 6. 컬럼 암호화 

컬럼 암호화(Column-Level Encryption)는 테이블의 특정 민감 컬럼만 선택하여 암호화하는 방식이다.

```
CUSTOMER 테이블

이름          평문 저장
주소          평문 저장
주민등록번호    암호화 저장
계좌번호       암호화 저장
```


### 장점
- 민감한 데이터만 선택하여 암호화할 수 있다.
- 사용자별로 세밀한 복호화 권한 설정이 가능하다.
- 전체 데이터베이스 암호화보다 암호화 범위가 작다.
- 의료·금융처럼 특정 민감정보를 보호하는 환경에 적합하다.

### 단점
- 어떤 컬럼을 암호화할지 별도로 분석해야 한다.
- 암호화된 컬럼의 데이터 길이가 증가할 수 있다.
- 인덱스 사용이 제한될 수 있다.
- 검색, 정렬, 범위 조회, 조인 성능이 저하될 수 있다.
- 응용프로그램이나 스키마 수정이 필요할 수 있다.
- 암·복호화가 빈번하면 시스템 부하가 증가한다.
  
  
# 7. 데이터베이스 전체 암호화 

데이터베이스 전체 암호화는 파일, 테이블스페이스 등 저장 단위 전체를 암호화하는 방식이다.

**TDE
```
DB 파일 또는 테이블스페이스
→ 전체 암호화
→ 디스크에 암호문 형태로 저장
```


### 장점
- 데이터 파일이 탈취되어도 원본 데이터를 직접 확인하기 어렵다.
- 모든 데이터를 암호화하므로 암호화 대상을 일일이 선정할 필요가 적다.
- 응용프로그램 수정이 거의 필요하지 않을 수 있다.
- 백업 파일과 저장매체 탈취 대응에 유리하다.

### 단점
- 민감하지 않은 데이터까지 암호화하여 시스템 부하가 증가할 수 있다.
- 사용자별로 세밀한 복호화 권한을 적용하기 어렵다.
- 정상적인 DB 권한을 가진 사용자의 조회를 제한하지 못한다.
- DBMS가 정상 동작 중이면 데이터가 자동으로 복호화된다.
- 키를 분실하면 전체 데이터베이스를 사용할 수 없게 될 수 있다.


# 8. 암호화 알고리즘 선정 

| 알고리즘 | 특징                          |
| ---- | --------------------------- |
| DES  | 실제 키 길이가 56비트로 짧아 취약        |
| 3DES | DES보다 안전하지만 처리 속도가 느림       |
| AES  | 128·192·256비트 키를 지원하며 널리 권장 |
| RSA  | 비대칭키 암호로 데이터키 보호·전달 등에 사용   |

# 9. 마스터키 

실제 데이터를 암호화하는 데이터키를 다시 암호화하여 보호하는 키 

```
데이터 → 데이터키로 암호화
데이터키 → 마스터키로 암호화
```

마스터키는 일반적으로 Wallet, Keystore, HSM 등 안전한 외부 저장소에 보관한다.


# 10. HSM 

HSM : Hardware Security Module

암호키를 안전한 하드웨어 장치 내부에서 생성,저장, 사용된다.

### 주요 기능
- 암호키 생성
- 암호키 저장
- 암호화·복호화 연산
- 키 외부 유출 방지
- 비인가 접근 차단
- 변조나 공격 감지 시 키 파기
- 자체 운영체제와 보안 기능 제공

### 장점
- 소프트웨어나 일반 파일보다 키를 안전하게 보호한다.
- 키가 장치 외부로 노출되는 위험을 줄인다.
- 강력한 물리적·논리적 보안을 제공한다.

### 단점
- 구축 비용이 높다.
- 장비 장애에 대비한 이중화가 필요하다.
- 운영과 연동이 복잡할 수 있다.


# 11. KMS 

KMS : Key Management System 

암호키의 전체 생명주기를 중앙에서 관리하는 시스템이다. 

### KMS의 주요 기능
- 키 생성
- 키 발급
- 키 저장
- 키 변경·회전
- 키 폐기
- 키 백업과 복구
- 키 접근통제
- 키 사용 이력 관리

### HSM과 KMS 비교

|구분|HSM|KMS|
|---|---|---|
|핵심 역할|하드웨어 내부에서 키 보호·연산|키 생명주기와 정책 관리|
|주요 목적|키 유출 방지|중앙집중적 키 운영|
|형태|보안 하드웨어 장치|관리 시스템·서비스|
|관계|KMS가 HSM을 이용할 수 있음|HSM과 연동하여 키를 보호할 수 있음|

> **HSM은 키를 안전하게 보관하고, 
> KMS는 키를 언제 만들고 변경하고 폐기할지 관리하는 시스템이다.**


# 12. TDE

TDE : Transparent Data Encription

DBMS가 데이터를 디스크에 저장할 때 자동으로 암호화하고, 메모리로 읽을 때 자동으로 복호화한다. 


# 13. Oracle TDE

```
원본 데이터
→ 테이블키 또는 테이블스페이스 키로 암호화
→ 데이터키를 TDE 마스터키로 암호화
→ 마스터키는 Wallet·Keystore·HSM에 저장
```

# 14. MySQL TDE

```
MySQL InnoDB
→ 데이터 암호화 키 사용
→ 키링 플러그인을 통해 마스터키 관리
→ 디스크에 암호화된 데이터 저장
```



# 15. Oracle TDE vs MySQL TDE

| 구분        | Oracle TDE                       | MySQL TDE           |
| --------- | -------------------------------- | ------------------- |
| 주요 암호화 단위 | 컬럼, 테이블스페이스                      | InnoDB 테이블스페이스 등    |
| 키 관리      | Wallet, Keystore, HSM, Key Vault | 키링 플러그인·외부 키 관리 서비스 |
| 실제 데이터 키  | 테이블·테이블스페이스 키                    | 테이블스페이스 데이터키        |
| 상위 보호 키   | TDE 마스터키                         | 마스터키                |
| 응용프로그램 수정 | 거의 필요 없음                         | 거의 필요 없음            |
| 주요 목적     | 저장 데이터 보호                        | 저장 데이터 보호           |

-> 
Oracle과 SQL Server의 TDE는 모두 저장된 데이터를 암호화하고, 사용자가 조회할 때 
DBMS 내부에서 자동으로 복호화한다는 점은 같다. 

Oracle TDE는 컬럼 암호화와 테이블스페이스 암호화를 지원하며, 실제 데이터를 암호화하는 테이블 키 또는 테이블스페이스 키를 마스터 키가 다시 암호화하여 보호한다. 즉, 데이터 암호화 키와 이를 보호하는 마스터 키가 분리된 구조이다. 
마스터 키는 Wallet, Keystore, Oracle Key Vault 또는 HSM에 저장할 수 있으며, 
HSM을 사용하면 키가 외부 보안 장비 안에서 생성·저장되어 유출 위험을 줄일 수 있다.

반면 SQL Server TDE는 주로 데이터베이스 단위로 데이터 파일과 로그 파일 전체를 암호화한다. 
실제 암호화에는 DEK, 즉 Database Encryption Key가 사용되며, 이 DEK는 인증서나 비대칭키로 보호된다. SQL Server에서 HSM을 사용할 경우에는 EKM을 통해 HSM과 연동하고, 
DEK를 보호하는 비대칭키를 HSM에 저장하는 방식으로 동작한다.

암호화 알고리즘 측면에서 Oracle은 AES, 3DES 등 다양한 알고리즘을 지원하며, 일반적으로 AES256이 권장된다. SQL Server도 AES_128, AES_192, AES_256, 3DES 계열을 지원하며, 보통 AES_256을 사용한다.

따라서 Oracle은 **마스터 키 기반으로 컬럼 또는 테이블스페이스를 암호화하는 방식**이고, 
SQL Server는 **DEK를 이용해 데이터베이스 전체를 암호화하고 인증서나 HSM 기반 비대칭키로 DEK를 보호하는 방식**이라고 정리할 수 있다.


# 16. 블록암호 운영모드 
|모드|주요 특징|주의점|
|---|---|---|
|ECB|각 블록 독립 암호화|패턴 노출, 권장하지 않음|
|CBC|이전 암호문과 연결|IV 필요, 오류가 두 평문 블록에 영향|
|CFB|블록 암호를 스트림처럼 사용|IV 필요|
|OFB|키 스트림 생성 후 평문과 결합|키 스트림·IV 재사용 금지|
|CTR|카운터값을 이용해 키 스트림 생성|병렬 처리 가능, 카운터 재사용 금지|
