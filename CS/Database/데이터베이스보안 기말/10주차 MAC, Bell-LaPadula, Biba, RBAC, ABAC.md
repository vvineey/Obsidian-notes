# 1. 강제적 접근 제어 (MAC)

MAC : Mandatory Access Control 
중앙관리자가 주체와 객체에 보안등급을 부여하고, 미리 정해진 보안규칙에 따라 접근을 강제 

### MAC 특징
- 주체와 객체에 보안등급을 부여 
- 중앙관리자가 보안정책을 관리 
- 객체 소유자도 임의로 권한을 변경 불가능 
- 기밀성이 중요한 군사/공공기관에서 사용 
- **규칙 기반 접근제어**라고도 함

### MAC 장점 
- DAC보다 보안성이 높음 
- 사용자가 타인에게 임의로 권한 전달 불가 
- 중앙에서 보안정책을 일관되게 적용 가능 
- 민감정보 유출을 강력하게 통제 가능 

### MAC 단점 
- 모든 주체와 객체에 등급을 지정해야 함 
- 정책 설계와 구현이 어려움 
- 관리 부담이 큼 
- 사용자의 자유로운 업무 수행 제한 
- 변화하 많은 상업 환경에 적용 어려움 


# 2. Bell-LaPadula 모델 

MAC 대표 모델로, 정보의 기밀성(Confidentiality) 보호가 목적이다.
높은 등급의 정보가 낮은 등급의 사용자나 객체로 유출되는 것을 방지한다.

	No Read Up, No Write Down


1. No Read Up
   보안등급이 낮은 주체는 보안등급이 높은 객체를 읽을 수 없다.
   
   ```
   읽기 조건 
   
   주체의 보안등급 ≥ 객체의 보안등급
   ```

2. No Write Down
   보안등급이 높은 주체는 낮은 등급의 객체에 쓸 수 없다.
   
```
쓰기 조건 

주체의 보안등급 ≤ 객체의 보안등급
```


# 3. Biba 모델

정보의 무결성 (Integrity) 보호가 목적이다.
신뢰도가 낮은 정보가 신뢰도가 높은 정보에 영향을 주는 것을 방지한다.

	No Read Down, No Write Up

1. No Read Down 
   무결성 등급이 높은 주체는 무결성 등급이 낮은 등급의 객체를 읽을 수 없다.
   신뢰도가 낮은 정보를 읽으면 높은 신뢰도 업무에 잘못된 정보가 사용될 수 때문
   
   ```
   읽기 조건 
   
   주체의 무결성 등급 ≤ 객체의 무결성 등급
   ```

2. No Write Up 
   무결성 등급이 낮은 주체는 무결성 등급이 높은 등급의 객체에 쓸 수 없다.
   
   ```
   쓰기 조건 
   
   주체의 무결성 등급 ≥ 객체의 무결성 등급
   ```

- 금융기관
- 회계 시스템
- 의료 시스템
- 중요 데이터 관리 시스템 

|구분|Bell-LaPadula|Biba|
|---|---|---|
|보호 목적|기밀성|무결성|
|읽기 규칙|No Read Up|No Read Down|
|쓰기 규칙|No Write Down|No Write Up|
|방지 대상|정보 유출|정보의 불법 변경·오염|
|대표 환경|군사·기밀 시스템|금융·회계 시스템|

- Bell-LaPadula: 기밀정보가 아래로 내려가면 안 됨
  → 위로 읽기 금지, 아래로 쓰기 금지

- Biba: 낮은 신뢰정보가 위로 올라가면 안 됨
  → 아래로 읽기 금지, 위로 쓰기 금지


# 4. Clark-Wilson 모델

상업 환경에서 데이터 무결성을 보호하기 위한 모델이다.
Biba가 보안등급을 이용해 무결성을 보호한다묜 
Clark-Wilson 모델은 올바르게 **정의된 거래**와 **임무분리**를 통해 무결성을 보호한다.

### 구성 요소 
1. CDI : 무결성이 보장되어야 하는 중요 데이터
2. UDI : 아직 신뢰 여부가 확인되지 않은 데이터 
3. TP : 데이터를 안전한 방식으로 변경하는 변환 절차 
4. IVP : 데이터 무결성을 검증하는 절차 


# 5. 역할기반 접근제어 (RBAC)

RBAC : Role-Based Access Control 
사용자에게 권한을 직접 부여하지 않고, **역할에 권한을 부여**한 뒤 사용자에게 역할을 배정한다.

```
사용자 -> 역할 -> 권한 -> 자원
```

### 구성 요소 
|요소|설명|
|---|---|
|사용자(User)|시스템이나 자원에 접근하는 대상|
|역할(Role)|업무나 직무를 기준으로 정의한 권한 집합|
|권한(Permission)|자원에 수행할 수 있는 작업|
|자원(Resource)|보호해야 하는 객체|
|세션(Session)|사용자가 역할을 활성화하여 작업하는 기간|

### 장점 
- 사용자에게 권한을 하나씩 부여하지 않아도 된다.
- 입사, 이동, 퇴사 시 역할만 변경하면 된다.
- 권한 부여와 회수가 효율적이다.
- 조직 구조와 업무를 반영하기 쉽다.
- 최소권한과 임무분리를 적용하기 쉽다.

### 단점 
- 역할 폭발 !!!! : 업무와 예외 상황이 많아지면 역할의 수가 지나치게 증가는 문제 
  ex ) 일반 사원, 인사팀 사원, 인사팀 야간 사원, 인사팀 계약직 야간 사원 ...

### 해결방안
- 역할을 업무 기준으로 재설계
- 중복 역할 통합
- 역할 계층 사용 
- 큰 권한 구조는 RBAC로 관리, 세부 조건은 ABAC로 관리 
- 역할과 권한을 정기적으로 검토 

# 6. RBAC 종류 

### 6.1 기본형 RBAC
사용자, 역할, 권한을 기본적으로 연결한다. 
- 사용자를 여러 역할에 배정 가능 
- 역할에 여러 권한 부여 가능 

### 6.2 계층형 RBAC
역할 사이에 상하 관계 설정하고 상위 역할은 하위 역할의 권한을 상속한다. 

	관리자 역할
	  └ 팀장 역할
	    └ 일반 사원 역할

### 6.3 제약형 RBAC 
계층형 RBAC + 임무 분리 제약 추가 

- 정적 임무분리
  한 사용자에게 충돌하는 역할을 동시에 부여하지 않는다.
  ex) 같은 사용자에게 구매 요청자와 구매 승인자 역할을 함께 부여 금지

- 동적 임무분리 
  두 역할을 부여할 수는 있지만 같은 세션에서 동시에 활성화할 수 없다.
  ex ) 사용자가 요청자와 승인자 역할을 모두 보유 → 한 업무에서는 두 역할을 동시에 사용할 수 없음

### 6.4 대칭형 RBAC
역할에 권한을 부여하는 관계를 관리하고 검토할 수 있도록 확장한 모델이다.
변화된 업무 상황에 맞게 권한과 역할의 관계를 관리하는데 중점 



# 7.  속성기반 접근 제어 (ABAC)

ABAC : Attribute-Based Access Control
주체와 객체가 가진 속성, 환경 조건, 접근제어 정책을 평가하여 접근 허용 여부를 결정한다.

```
사용자의 부서가 심장내과이고,
사용자가 해당 환자의 담당의이며,
현재 시간이 업무시간이고,
병원 내부 네트워크에서 접속한 경우에만
환자 기록 조회 허용
```

### 주요 요소 
- 주체 속성
- 객체 속성
- 환경 조건 
- 접근제어 정책

### 동작 과정 
1. 주체가 객체에 접근을 요청한다.
2. 시스템이 접근제어 정책을 확인한다.
3. 주체 속성, 객체 속성, 환경 조건을 수집한다.
4. 속성과 조건이 정책을 만족하는지 평가한다.
5. 접근 허용 또는 거부를 결정한다.
6. 결정 결과를 집행한다.

### 장점 
- 시간, 위치 등 동적인 환경 조건을 반영 가능 
- 주체와 객체의 속성을 이용해 세밀한 접근제어 가능 
- 역할을 계속 만들지 않아도 됨 
- 클라우드, IoT, 의료 등 변화가 빠른 환경에 적합 


### 단점 
- 속성을 정확하고 최신 상태로 유지해야 함 
- 속성이나 정책 설정 오류 -> 보안 문제 
- 접근 요청마다 여러 속성 평가 -> 성능 비용 문제 
- 정책 많음 -> 관리/ 검증 복잡




# 8. RBAC vs ABAC

| 구분    | RBAC            | ABAC                   |
| ----- | --------------- | ---------------------- |
| 접근 기준 | 역할              | 주체·객체 속성 및 환경 조건       |
| 정책 예시 | 의사 역할이면 조회 허용   | 담당 의사가 업무시간에 접속할 때만 허용 |
| 장점    | 권한 관리가 간단하고 효율적 | 동적이고 세밀한 통제 가능         |
| 단점    | 역할 폭발 가능        | 속성·정책 관리가 복잡           |
| 적합 환경 | 조직 구조가 명확한 환경   | 클라우드·IoT·분산 환경         |
