

# 1. 접근제어 

정보보안 정책에 따라 허가된 사용자만 시스템 자원에 접근하도록 제한하는 것 

-> 데이터베이스는 외부 공격뿐 아니라 **권한을 가진 내부 사용자의 권한 남용**도 관리해야 함 


# 2. 접근제어 수행 요소 

### 주체 (Subject)
객체에 접근하려는 대상 

- 사용자 / 사용자 그룹
- 응용프로그램
- 실행 중인 프로세스 
- 데이터베이스 계정 

### 객체 (Object)
주체가 접근하려는 보호 대상 

- 데이터베이스 
- 테이블
- 컬럼 
- 파일
- 서버 

### 접근권한 (Privilege)
주체가 객체에 수행할 수 있는 행위

|기호|권한|설명|
|---|---|---|
|R|Read|객체의 내용을 읽을 수 있음|
|W|Write|객체의 내용을 수정할 수 있음|
|X|Execute|프로그램이나 프로시저를 실행할 수 있음|
|D|Delete|객체나 데이터를 삭제할 수 있음|
|C|Create|객체를 생성할 수 있음|



# 3. 접근제어 절차 

	식별 -> 인증 -> 인가 

### 식별 (Identification)
사용자가 시스템에 자신이 누군인지 알리는 과정

### 인증 (Authentication)
사용자의 신원이 실제로 맞는지 검증하는 관정 

| 방식    | 예                  |
| ----- | ------------------ |
| 지식 기반 | 비밀번호, PIN          |
| 소유 기반 | 스마트카드, 보안 토큰, 휴대전화 |
| 생체 기반 | 지문, 얼굴, 홍채         |
### 인가 (Authorization)
인증된 사용자가 특정 객체에 대해 어떤 행위를 수행할 수 있는지 결정하는 과정 



# 4. 데이터베이스 접근제어 규칙 설계
접근제어 규칙은 *명칭부 + 조건부 + 제어부* 로 구성된다.

### 명칭부 
접근제어 규칙 이름으로, 관리자가 규칙의 목적과 대상을 쉽게 알 수 있도록 작성해야 함 

```
급여 테이블 대량조회 경보 규칙
김보빈 HRDB 로그인 규칙
```

### 조건부 
접근제어 규칙이 적용되는 상황을 정의한다.
IP주소나 MAC 주소는 변경되거나 위조될 수 있으므로 이것만으로 사용자를 신뢰하면 안 됨 

| 조건 종류    | 예                   |
| -------- | ------------------- |
| 사용자 속성   | 사용자 ID, IP 주소       |
| 정보 속성    | DB명, DB 계정, 테이블, 컬럼 |
| 접속 수단 속성 | 접속 도구, 단말기명, 프로토콜   |
| 환경 조건    | 접속 날짜와 시간           |

```
사용자 ID가 VVIN이고 
IP 주소가 192.168.~~이고 
HTDB1 데이터베이스에 HR_USER 계정에 접속하는 경우 
```


### 제어부 
조건이 만족되었을 때 시스템이 어떤 통제를 수행할지 정의

- 로그인 통제
  데이터베이스 로그인 허용 여부를 결정
  -> 특정 IP, 업무시간, 특정 접속 도구, 중요 계정의 외부 접속 차단 등 
  
- SQL 수행 통제
  SQL을 실행할 때 권한을 확인
  -> 특정 테이블 조회, 특정 칼럼 조회, DROP, ALTER TABLE 차단 등 
  
- 로깅 통제
  실행한 SQL과 접근 정보를 기록 
  
- 경보 통제
  보안정책 위반 발생 시 관리자에게 알림 


# 5. 우회 접근과 차단 

## 우회접근 
사용자가 접근제어 서버를 거치지 않고 DB 서버에 직접 접속하는 것 

## 우회접근 차단 방법 

### 방법 1. 프록시 게이트웨이 방식
사용자가 DB에 접속할 때 프록시 서버를 반드시 거치도록 함 

- 장점 
  - 중앙에서 로그인/ SQL 통제 가능 
  - 접근 기록 관리 쉬움

- 단점 
  - 프록시 서버 장애 -> 서비스에 영향
  - 사용자가 접속 설정을 변경하여 직접 접속할 가능성 있음 

###  방법 2. 에이전트 방식
DB 서버에 접근제어 에이전트를 직접 설치하여 콘솔을 통한 직접 접근도 통제 

- 장점 
  - 직접 DB 접속을 효과적으로 차단 가능 

- 단점 
  - DB 서버에 프로그램 설치 -> 서버 성능과 안정성에 영향을 줄 수 있음 

### 방법 3. 서버 보안 방식
운영체제 커널이나 네트워크 포트 수준에서 DB 접근권한을 확인 
ex) 접근제어 서버 IP에서만 들어오는 요청만 DB 포트에 접속하도록 설정 


### 방법 4. 스니핑 방식
네트워크 패킷을 복사하여 사용자 로그인과 SQL 실행을 감시함, 권한이 없는 접근 발견 -> 세션 종료 

- 장점 
  - DB 서버에 직접 프로그램을 설치하지 않아도 됨 

- 단점 
  - 차단 기능이 오작동 -> 정상 업무 세선까지 종료할 수 있음 


# 6. 임의적 접근제어 (DAC)

DAC : Discretionary Access Control
객체의 소유자가 자신의 판단에 따라 다른 사용자에게 권한을 부여하거나 회수한다.

```
Z가 보고서 파일의 소유자
→ Z가 B에게 읽기 권한 부여
→ Z가 B의 읽기 권한 회수
```

사용자의 신분을 기준으로 접근을 통제하므로 **신분 기반 접근제어**라고도 한다.

장점 
- 구현 간단 
- 사용하기 쉽고 유연 
- 객체별로 세부적 권한 설정 
- 객체 소유자가 필요에 따라 빠르게 권한 변경 

단점 
- 잘못된 사용자에게 권한 부여 가능성
- 권한이 다른 사용자에게 계속 전파될 가능성 
- 계정 탈취 -> 사용자가 가진 권한도 탈취 
- 객체가 많아지면 권한 관리 복잡
